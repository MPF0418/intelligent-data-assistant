# V3.0 智能路由架构说明

**架构版本**: V3.0  
**设计时间**: 2026-02-28  
**架构目标**: 根据用户需求的特征，智能选择最合适的处理引擎

---

## 一、架构概述

### 1.1 设计思想

传统架构的问题是线性的 if-else 判断，无法根据需求的复杂程度灵活选择处理引擎。V3.0 采用**智能路由**架构，通过需求分类模块分析用户输入的特征，动态选择：

- **精准模式**（本地模型）：用户提供了准确的列名和清晰的指令
- **智能模式**（大模型）：用户表达模糊、列名不明确、有复杂需求

## 二、核心模块

### 2.1 需求分类模块 (`requirementClassifier.js`)

**职责**: 分析用户输入，决定使用哪种处理模式

**核心方法**:

#### `classify(userInput, columns, sampleData)`

**输入**:
- `userInput`: 用户输入文本
- `columns`: 数据表列名数组
- `sampleData`: 样本数据（用于推断字典值）

**输出**:
```javascript
{
  mode: 'precise' | 'intelligent',
  confidence: 0.95,
  reason: '分类理由',
  matchedColumns: [
    { column: '省公司名称', matchType: 'exact' },
    { column: '险情确认时长', matchType: 'partial' }
  ],
  ambiguityScore: 0.2,
  hasComplexRequirements: true,
  columnMatchScore: 0.85
}
```

**分类逻辑**:

1. **复杂需求检测** (优先级最高)
   - 单位转换关键词："转换"、"转成"、"单位为"
   - 数据预处理关键词："保留X位小数"、"格式化"
   - 复杂计算关键词："增长率"、"占比"、"同比环比"
   - 复杂筛选关键词："大于"、"小于"、"并且"、"或者"
   - **结果**: 只要有复杂需求，强制使用智能模式

2. **模糊度评分** (0-1)
   - 未匹配到列名：+0.5
   - 部分匹配列名：+0.2 × 部分匹配数
   - 模糊词汇（什么、哪些、怎么）：+0.1 × 词汇数
   - 非结构化表达：+0.2
   - **阈值**: > 0.3 使用智能模式

3. **列名匹配度** (0-1)
   - 完全匹配：1.0
   - 部分匹配：0.5
   - **阈值**: < 0.8 使用智能模式

#### `getContextForLLM(columns, sampleData, mentionedColumns)`

**职责**: 为大模型构建上下文信息

**输出**:
```javascript
[
  {
    name: '省公司名称',
    type: 'categorical',
    sampleValues: ['安徽区域', '江苏区域', '京津冀区域'],
    isMatched: true
  },
  {
    name: '险情确认时长',
    type: 'numeric',
    sampleValues: [120, 300, 1800],
    isMatched: true
  }
]
```

### 2.2 大模型提示词模块 (`llmPrompts.js`)

**职责**: 为大模型提供结构化的提示词模板

**核心方法**:

#### `generateChartConfigPrompt(userInput, columnInfo, dataInfo)`

**适用场景**: 用户需要生成图表

**提示词结构**:
1. **角色定义**: "你是一位数据可视化专家"
2. **用户输入**: 原始输入文本
3. **数据结构**: 列名、类型、示例值、是否被用户提到
4. **任务要求**: 理解需求、选择列、确定聚合、处理转换
5. **输出格式**: JSON Schema 定义
6. **规则说明**: chartType、xAxisColumn、yAxisColumn 等字段的详细说明
7. **示例**: 完整的输入输出示例

**输出示例**:
```json
{
  "chartType": "bar",
  "xAxisColumn": "省公司名称",
  "yAxisColumn": "险情确认时长",
  "aggregateFunction": "avg",
  "title": "各省公司险情确认时长平均值",
  "dataTransform": {
    "unitConversion": {
      "from": "秒",
      "to": "分钟",
      "formula": "秒数/60"
    },
    "decimalPlaces": 2
  }
}
```

#### `generateQueryConfigPrompt(userInput, columnInfo, dataInfo)`

**适用场景**: 用户需要查询数据

**输出示例**:
```json
{
  "queryType": "filter",
  "filters": [
    {
      "column": "险情确认时长",
      "operator": "greater",
      "value": 1800
    }
  ],
  "sortBy": {
    "column": "险情确认时长",
    "order": "desc"
  }
}
```

#### `generateIntentRecognitionPrompt(userInput, columnInfo)`

**适用场景**: 判断用户是要查询还是生成图表

## 三、处理流程

### 3.1 精准模式流程

```
用户输入: "按照省公司名称统计险情确认时长平均值"
    ↓
需求分类模块
    ↓
提取列名:
  - "省公司名称" → 完全匹配
  - "险情确认时长" → 完全匹配
    ↓
计算匹配度: 1.0 (完全匹配)
检测复杂度: false (无复杂需求)
模糊度评分: 0.1 (低模糊度)
    ↓
决策: 精准模式
    ↓
本地意图识别器
    ↓
规则匹配 → BERT模型验证
    ↓
生成配置:
  {
    intent: 'CHART_BAR',
    groupCol: '省公司名称',
    valueCol: '险情确认时长',
    aggregateFunction: 'avg'
  }
    ↓
执行图表生成
```

### 3.2 智能模式流程

```
用户输入: "按照省公司的险情确认时长平均值（需要转换成分钟，保留2位小数）绘制柱状图"
    ↓
需求分类模块
    ↓
提取列名:
  - "省公司" → 部分匹配("省公司名称")
  - "险情确认时长" → 完全匹配
    ↓
计算匹配度: 0.75 (部分匹配)
检测复杂度: true (检测到单位转换、保留小数)
模糊度评分: 0.35 (中等模糊度)
    ↓
决策: 智能模式
    ↓
构建上下文:
  - 列信息（含类型、示例值）
  - 标记用户提到的列
    ↓
大模型提示词生成
    ↓
调用大模型API
    ↓
解析返回的JSON配置
    ↓
执行图表生成（含数据转换）
```

## 四、关键特性

### 4.1 智能降级

```
精准模式失败
    ↓
自动切换智能模式
    ↓
使用大模型处理
```

### 4.2 上下文感知

大模型接收的上下文包括：
- **列名**: 完整列名列表
- **类型**: 每列的数据类型（数值、分类、时间）
- **示例值**: 每列的样本值（帮助理解列的含义）
- **匹配标记**: 标记用户提到的列

### 4.3 复杂需求处理

支持的复杂需求：
- ✅ 单位转换（秒→分钟、元→万元）
- ✅ 数据格式化（保留小数、千分位）
- ✅ 复杂计算（增长率、占比、同比环比）
- ✅ 多条件筛选（并且、或者）
- ✅ 数据预处理（过滤、排序、分组）

## 五、性能优化

### 5.1 响应时间

| 模式 | 平均响应时间 | 适用场景 |
|------|-------------|----------|
| 精准模式 | < 100ms | 简单查询、明确列名 |
| 智能模式 | 2-5s | 复杂需求、模糊表达 |

### 5.2 Token 优化

- 精准模式：0 Token（本地处理）
- 智能模式：~2000 Token（含上下文信息）

### 5.3 缓存策略

- 相同输入的意图识别结果缓存 5 分钟
- 列名匹配结果缓存（数据表不变时）

## 六、错误处理

### 6.1 降级策略

```
智能模式超时 (45s)
    ↓
自动降级到精准模式
    ↓
使用本地模型处理
    ↓
如果仍失败，返回错误提示
```

### 6.2 错误类型

| 错误类型 | 处理方式 |
|---------|----------|
| 大模型超时 | 降级到本地模型 |
| 大模型返回格式错误 | 尝试解析，失败则降级 |
| 本地模型拒识 | 提示用户明确需求 |
| 列名无法匹配 | 提示可用的列名 |

## 八、总结

V3.0 智能路由架构的优势：

1. **智能化**: 根据需求特征自动选择最优处理引擎
2. **高效性**: 简单需求用本地模型快速响应，复杂需求用大模型智能处理
3. **准确性**: 大模型获得完整的上下文信息，列名匹配更准确
4. **可扩展性**: 易于添加新的分类规则和提示词模板
5. **容错性**: 自动降级机制确保系统稳定性

**新架构让系统既快又准，既智能又稳定！**
