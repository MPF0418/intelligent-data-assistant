# 架构反思与优化

**反思时间**: 2026-02-28  
**核心问题**: 
1. 智能模式超时（两次大模型调用）
2. Skills 未被调用

---

## 一、问题分析

### 1.1 超时问题

**现象**:
```
[20:01:43] 进入智能模式（大模型）
[20:01:58] 意图识别完成 - 耗时: 15021ms (15.02秒)
[20:01:58] 处理失败 - 请求超时
```

**根因**:
```javascript
// 错误的实现：两次大模型调用
const intentPrompt = window.llmPrompts.generateIntentRecognitionPrompt(...);
const intentResponse = await callLLMAPI(intentPrompt, signal, 15000);  // 第一次

if (parsedIntent.intent === 'chart') {
    llmPrompt = window.llmPrompts.generateChartConfigPrompt(...);
} else {
    llmPrompt = window.llmPrompts.generateQueryConfigPrompt(...);
}

const llmResponse = await callLLMAPI(llmPrompt, signal, 45000);  // 第二次
```

**问题**:
1. 两次调用增加了失败概率
2. 第一次超时后第二次无法执行
3. 总耗时 = 15s + 45s = 60s，用户体验差

### 1.2 Skills 未被调用

**现象**:
- 系统有 6 个 Skills（数据清洗、数据分析、趋势分析、异常检测、业务建议、图表生成）
- 但从未在日志中看到 Skills 被调用

**根因**:
1. 需求分类模块没有判断是否需要调用 Skills
2. 处理流程中没有集成 Skills 调用逻辑
3. Skills 和主流程是割裂的

---

## 二、优化方案

### 2.1 超时问题修复

**方案**: 只调用一次大模型，直接生成配置

```javascript
// 优化后：一次大模型调用
const hasChartKeyword = /图|表|可视化|绘制|画/.test(userInput);
const hasQueryKeyword = /查找|查询|筛选|统计/.test(userInput);

if (hasChartKeyword || !hasQueryKeyword) {
    llmPrompt = window.llmPrompts.generateChartConfigPrompt(...);
} else {
    llmPrompt = window.llmPrompts.generateQueryConfigPrompt(...);
}

const llmResponse = await callLLMAPI(llmPrompt, signal, 60000);  // 只调用一次
```

**优化点**:
1. ✅ 通过关键词判断意图，不需要单独调用大模型
2. ✅ 只调用一次大模型生成配置
3. ✅ 超时时间增加到 60 秒（给大模型足够时间）
4. ✅ 减少了 50% 的大模型调用次数

### 2.2 Skills 集成方案

**方案**: 在需求分类阶段判断是否需要调用 Skills

```javascript
// 新增：检测需要调用的Skills
detectRequiredSkills(text, columns) {
    const requiredSkills = [];
    
    // 1. 数据清洗 Skill
    if (/清洗|清理|去重|填充|缺失值/.test(text)) {
        requiredSkills.push({ name: 'dataCleaning', priority: 1 });
    }
    
    // 2. 数据分析 Skill
    if (/分析|统计|指标|洞察/.test(text)) {
        requiredSkills.push({ name: 'dataAnalysis', priority: 2 });
    }
    
    // 3. 趋势分析 Skill
    if (/趋势|走势|变化|增长|同比|环比/.test(text)) {
        requiredSkills.push({ name: 'trendAnalysis', priority: 3 });
    }
    
    // 4. 异常检测 Skill
    if (/异常|异常值|离群|偏离/.test(text)) {
        requiredSkills.push({ name: 'anomalyDetection', priority: 4 });
    }
    
    // 5. 业务建议 Skill
    if (/建议|推荐|优化|改进|方案/.test(text)) {
        requiredSkills.push({ name: 'businessAdvice', priority: 5 });
    }
    
    // 6. 图表生成 Skill
    if (/图|表|可视化|绘制/.test(text)) {
        requiredSkills.push({ name: 'chartGenerator', priority: 6 });
    }
    
    return requiredSkills;
}
```

**集成到决策逻辑**:
```javascript
// 如果需要调用分析类Skills，优先使用智能模式
const analysisSkills = requiredSkills.filter(s => 
    ['dataAnalysis', 'trendAnalysis', 'anomalyDetection', 'businessAdvice'].includes(s.name)
);
if (analysisSkills.length > 0) {
    mode = 'intelligent';
    reasons.push(`需要调用分析类Skills: ${analysisSkills.map(s => s.name).join(', ')}`);
}
```

## 三、新架构流程

```
用户输入
    ↓
需求分类模块
    ├─ 提取列名（完全匹配、部分匹配）
    ├─ 计算匹配度评分
    ├─ 检测模糊度
    ├─ 检测复杂需求
    ├─ 检测需要调用的Skills ← 新增
    ↓
分类决策
    ├─ 有复杂需求 → 智能模式
    ├─ 模糊度高 → 智能模式
    ├─ 列名匹配度低 → 智能模式
    ├─ 需要调用Skills → 智能模式 ← 新增
    └─ 否则 → 精准模式
    ↓
    ├─ 精准模式（本地模型）
    │       ├─ 规则匹配
    │       ├─ BERT模型验证
    │       └─ 生成配置
    │
    └─ 智能模式（大模型）
            ├─ 检查是否需要执行Skills ← 新增
            ├─ 构建上下文（列信息、示例值）
            ├─ 生成优化提示词
            ├─ 调用大模型（一次） ← 优化
            ├─ 解析配置
            └─ 执行Skills（如果需要） ← 新增
    ↓
配置执行引擎
    ├─ 查询执行
    ├─ 图表生成
    ├─ 数据转换
    └─ Skills执行 ← 新增
```

## 七、总结

### 核心改进

1. **性能提升**: 大模型调用次数从 2 次减少到 1 次，响应时间从 15s+ 降低到 5-10s
2. **功能增强**: Skills 在需求分类阶段就被识别和调用，不再割裂
3. **架构优化**: 需求分类模块成为真正的"智能路由中心"，统一决策

### 架构价值

- **智能化**: 根据需求特征自动选择最优处理路径
- **高效性**: 减少不必要的大模型调用，降低成本
- **完整性**: Skills 被正确识别和调用，功能完整
- **可扩展性**: 易于添加新的 Skills 和分类规则

**新架构让系统更快、更智能、更完整！**
