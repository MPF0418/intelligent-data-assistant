## 1. 需求背景

### 1.1 业务背景

在政务数据分析领域，业务人员经常需要处理大量的结构化数据（如应急事件记录、人员信息、财务数据等），传统数据分析方式存在以下痛点：

1. **技术门槛高**：传统数据分析工具（如Excel高级功能、SQL、Python等）需要专业技能，普通业务人员难以掌握
2. **查询效率低**：业务人员需要将数据需求转达给技术人员，沟通成本高，响应周期长
3. **可视化困难**：图表制作需要专业知识，无法快速将数据洞察转化为可视化呈现
4. **分析深度不足**：缺乏智能分析能力，难以发现数据中的隐藏规律和异常情况
5. **报告生成繁琐**：数据分析报告需要人工编写，耗时耗力

### 1.2 用户画像

| 用户类型 | 特征描述 | 核心需求 |
|----------|----------|----------|
| 政务业务人员 | 熟悉业务逻辑，编程基础薄弱 | 自然语言查询数据、快速生成图表、获取业务建议 |
| 数据分析师 | 具备数据分析能力，需要高效工具 | 数据清洗、统计分析、趋势预测、异常检测 |
| 决策管理者 | 关注数据洞察和决策支持 | 数据概览、关键指标、业务建议、报告导出 |

### 1.3 解决方案

智能数据洞察助手是一款基于AI的数据分析可视化工具，通过自然语言交互方式，让用户无需编程即可完成数据查询、分析、可视化等操作，大幅降低数据分析门槛，提升工作效率。

**V4.0核心升级**：
- **三层意图识别架构**：规则匹配 + 本地BERT模型 + 大模型API兜底，响应时间从2-5秒降至<50ms
- **本地查询配置生成**：无需大模型API即可生成查询/图表配置，响应时间<10ms
- **SQL.js数据库模式**：支持大数据量处理，自动切换内存/数据库模式
- **智能拒识机制**：对无效输入进行拒识，提供友好提示
- **智能路由架构（V3.0）**：需求分类器自动选择精准模式或智能模式，优化处理效率
- **实体提取与链接（V4.0）**：增强实体识别能力，支持matchCount机制验证实体链接有效性
- **filter_aggregate查询优化（V4.0）**：修复带筛选条件的聚合查询路由问题，确保正确执行
- **Agent路由器修复（V4.0）**：修复Agent调度器绕过传统查询处理的问题，保证查询正确执行

## 2. 需求目标

### 2.1 业务目标

| 目标 | 描述 | 衡量指标 |
|------|------|----------|
| 降低使用门槛 | 让非技术人员能够独立完成数据分析 | 用户培训时间 < 30分钟 |
| 提升查询效率 | 通过自然语言快速查询数据 | 本地意图识别响应 < 50ms |
| 增强可视化能力 | 一键生成专业图表 | 图表生成成功率 > 95% |
| 提供智能洞察 | AI自动分析数据并给出建议 | 分析报告准确率 > 90% |
| 大数据支持 | 支持大数据量文件处理 | 支持100万行+数据 |

### 2.2 功能目标

1. **数据导入**：支持CSV、Excel格式文件上传，自动解析数据结构
2. **智能查询**：用户通过自然语言描述查询需求，AI自动理解并执行
3. **智能绘图**：用户通过自然语言描述可视化需求，AI自动生成图表
4. **数据筛选**：支持多字段叠加筛选，灵活过滤数据
5. **AI分析报告**：自动生成数据分析报告，支持导出PDF/图片
6. **处理日志**：详细记录处理过程，便于问题排查和性能优化
7. **技能扩展**：支持模块化技能扩展，增强系统能力
8. **大数据处理**：SQL.js数据库模式，支持大数据量查询
9. **设置管理**：支持配置本地意图识别、数据库模式等选项

### 2.3 成功标准

- 用户能在5分钟内完成从数据上传到获取分析结果的全流程
- 系统支持100MB以内的数据文件处理
- 本地意图识别准确率 > 90%（规则+本地模型）
- 整体意图识别准确率 > 95%（含API兜底）
- 系统可用性 > 99%

| 性能 | 本地配置生成 < 10ms | 无需API调用 |
| 性能 | 数据库查询 < 100ms | SQL.js模式 |

- 后端服务部署（纯前端实现，意图识别API为本地服务）

- 数据持久化存储（除IndexedDB缓存外）
- 实时数据接入
- 多人协作功能

## 14. 实体提取与链接系统 (V4.0新增)

### 14.1 系统概述

实体提取与链接系统是V4.0版本的核心升级，用于从用户输入中提取关键实体（如地区名、产品名、数值等），并将其链接到数据表的对应列。

### 14.2 实体类型定义

| 实体类型 | 描述 | 示例 |
|----------|------|------|
| LOCATION | 地区/位置实体 | 广东、北京、华东区域 |
| PRODUCT | 产品实体 | 产品A、产品B |
| PERSON | 人员实体 | 张三、李四 |
| ORGANIZATION | 组织实体 | 销售部、研发中心 |
| NUMBER | 数值实体 | 100、50万 |
| DATE | 日期实体 | 2024年、3月份 |
| COLUMN_REF | 列名引用 | 销售额、数量 |

### 14.3 实体链接机制

**matchCount机制**：验证实体链接有效性的核心指标

```javascript
// 实体链接结果示例
{
    text: "广东",
    type: "LOCATION",
    linkedColumn: "省份",
    matchCount: 15,  // 数据中匹配到的记录数
    confidence: 0.95
}

// 路由决策逻辑
if (matchCount > 0) {
    // 实体链接有效，使用精准模式
    routeToPreciseMode();
} else {
    // 实体链接失败，使用智能模式
    routeToIntelligentMode();
}
```

### 14.4 列名匹配规则

V4.0修复了列名匹配的问题，避免"产品A"被误识别为列名"产品"：

```javascript
// 匹配规则（按优先级）
// 1. 精确匹配：text === header
// 2. 列名包含text：header.includes(text) && text.length >= 2
// 注意：不再使用 text.includes(header)，避免误匹配
```

### 14.5 filter_aggregate查询处理

V4.0新增对filter_aggregate查询类型的完整支持：

| 查询类型 | 描述 | 示例输入 |
|----------|------|----------|
| filter_aggregate | 带筛选条件的聚合查询 | "广东的销售额是多少"、"产品A的总数量" |

**处理流程**：
1. 实体提取：识别筛选值（如"广东"）
2. 实体链接：将筛选值链接到对应列
3. 配置生成：生成filter_aggregate配置
4. 路由判断：确保不走Agent调度器，直接执行本地查询
5. 执行查询：先筛选再聚合

### 14.6 Agent路由器绕过修复

V4.0修复了Agent调度器绕过传统查询处理的问题：

```javascript
// 修复前：Agent调度器会拦截所有查询
if (window.agentRouter) {
    // Agent处理，可能导致filter_aggregate被错误处理
}

// 修复后：filter_aggregate查询跳过Agent调度器
if (window.agentRouter && enhancedConfig.queryType !== 'filter_aggregate') {
    // 非filter_aggregate查询才走Agent调度器
}
```
